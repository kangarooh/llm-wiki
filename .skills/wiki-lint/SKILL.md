# wiki-lint 技能：知识图谱健康巡检

## 核心目标
将软件工程中的"静态代码分析"引入知识管理。定期运行此 skill，找出知识库长期演进中产生的：死链、孤岛、未同步索引、认知冲突。

## 触发条件
- 用户输入 `/wiki-lint`
- 用户询问"我的知识库健康状况如何"
- 用户要求"检查知识库状态"或"检查健康"

## 巡检流水线

### 第 1 步：数据获取

1. 获取 `wiki/` 下所有 `.md` 文件列表。
2. 读取 `wiki/index.md`，提取当前已注册在案的页面清单。

### 第 2 步：一致性与拓扑健康分析

1. **索引同步检查**：将实际文件列表与 `index.md` 提取的清单比对，找出"存在但未注册"或"已注册但文件丢失"的页面。
2. **死链定位**：优先使用 `obsidian unresolved` 命令直接获取死链列表。如果 obsidian-cli 不可用，回退为手动遍历 wiki 页面提取 `[[双链]]` 引用并检查目标是否存在。
3. **孤儿页面识别**：优先使用 `obsidian orphans` 命令直接获取孤儿页面列表。如果 obsidian-cli 不可用，回退为手动查找无入链引用的页面（排除 `index.md` 和 `log.md`）。

### 第 3 步：认知冲突提取

搜索所有包含 `## 知识冲突` 标题的 `.md` 文件，提取冲突内容。

### 第 4 步：格式检查

检查所有页面是否符合 `llm-wiki` skill 规范：
- 是否有 YAML frontmatter
- 是否包含 `## 关联连接` 区域
- 文件名是否符合命名规范

## 报告输出规范

扫描完成后，输出结构化报告，严格遵循以下格式：

```
## 知识库健康体检报告 — YYYY-MM-DD

### 绿灯项
- [运行良好的项目]

### 黄灯项
- **发现 N 个孤儿页面**：
  - `[[页面A]]`（建议关联到 `[[某分类]]`）
  - `[[页面B]]`
- **发现 N 个未同步索引**：[列表] — 文件存在但未在 index.md 注册

### 红灯项
- **发现 N 个死链**：
  - [来源页面] → 引用了不存在的 `[[目标页面]]`
- **存在 N 个未解决的知识冲突**：
  - `[[页面名称]]` — 冲突摘要：A 观点 vs B 观点

### 下一步行动
1. 回复「修复索引」：自动将未同步页面追加到 index.md
2. 回复「修复死链」：批量建立空白占位页或移除无效引用
3. 回复「推演冲突」：针对特定页面生成知识冲突调和草案
```

## 硬约束
- **仅读扫描**：生成报告前，禁止修改、删除、重命名任何文件
- **手动确认**：报告后等待用户确认再执行修复
- **记录日志**：用户确认并完成修复后，在 `wiki/log.md` 追加操作日志：

```markdown
## [YYYY-MM-DD] wiki-lint | 修复了 N 个问题
- **处理清单**: [具体修复内容]
```
